Actualizacion 2026-06-06 - fix "Activar inferencia" se queda cargando
----------------------------------------------------------------------
Se diagnostico y corrigio el bug donde hacer clic en "Activar inferencia" en el
dashboard deja el video cargando indefinidamente sin mostrar el stream con IA.

Causa raiz (dos capas):
1. API central no tiene los paths /INFERENCE registrados en stream_paths DB.
   Al consultar /streams/CAM-ID%2FINFERENCE/status, API central devuelvia 403.
   El dashboard interpretaba el 403 como "stream no disponible" y devolvio
   online=false al frontend, que mostraba INFERENCE_LOADING_SRCDOC sin fin.

2. La URL del player de INFERENCE generada por el bypass no tenia trailing slash.
   Esto causaba que MediaMTX devolviera un redirect 301 al cargar el player.
   Durante ese redirect, la primera sesion WebRTC (WHEP POST->201) tenia el
   PATCH de ICE rechazado con 404, rompiendo la negociacion WebRTC.
   La segunda sesion funcionaba (PATCH->204) porque el frame se recargaba,
   pero eso agrego un retardo de ~4 segundos y dependia de un segundo intento.

Archivo modificado:
- /root/robiotec/dashboard/back/app/application.py

Cambio 1 - bypass INFERENCE en _authorized_camera_viewer:

     GET /v3/paths/get/{path_encoded}
   Si source.type existe (stream publicando), se devuelve online=true con una
   viewer_url construida desde settings.mediamtx_webrtc_base_url.
   Esto evita completamente la consulta a API central para paths de inferencia.

Cambio 2 - trailing slash en viewer_url del bypass INFERENCE:
   La viewer_url generada para INFERENCE ahora incluye trailing slash:
     /mediamtx/CAM-MPCU71OU-0787AE/INFERENCE/
   Antes era:
     /mediamtx/CAM-MPCU71OU-0787AE/INFERENCE  (sin slash final)
   El slash final es necesario porque MediaMTX sirve el player HTML en /{path}/
   (con trailing slash) y envia un 301 si se pide sin el slash. Ese 301 creaba
   una condicion de carrera donde la sesion WHEP recien creada era invalidada
   antes de que el browser enviara el PATCH de ICE trickle, dando 404.
   Con el slash, el player carga directo y WHEP funciona de forma consistente.

Actualizacion 2026-06-06 - Causa raiz definitiva del "cargando infinito" INFERENCE
===========================================================================

PROBLEMA RAIZ CONFIRMADO (con prueba directa):
MediaMTX v1.17.1 ELIMINA todas las sesiones WebRTC/WHEP de un path cuando
el SRT source se desconecta. Esto es el comportamiento por defecto de MediaMTX
con paths de tipo "publisher".

Secuencia de fallo:
1. Usuario activa inferencia -> se llama camera-preview-frame?exclusive=1&inference=1
2. Esto llama _authorized_camera_viewer(inference=True) que escribe en
   camera_inference_view_requests via _set_camera_inference_view_request
3. El orchestrador en 10.0.0.2 detecta el view request, mata el ffmpeg preview
   (480x270@1fps) y arranca ffmpeg view (1280x720@8fps) -> SRT SE CAE ~1-2 SEGUNDOS
4. MediaMTX ve que el SRT source desaparecio -> MATA la sesion WHEP que estaba
   negociando ICE (aunque pce=False, la sesion desaparece del todo)
5. El browser tiene ICE en estado "checking" -> como ya no recibe respuesta STUN
   de MediaMTX -> ICE timeout a los ~18s -> reader.js DELETE->404 (ya no existe)
6. ADEMAS: durante la caida SRT, camera-viewer-url?inference=1 retorna online=false
   -> startInferencePolling() pone frame.srcdoc=INFERENCE_LOADING_SRCDOC
7. Siguiente poll con online=true -> llama camera-preview-frame?exclusive=1 (KICK!)
   -> mata la sesion del browser que estaba retrying -> crea nueva sesion
8. Vuelve al paso 3 -> loop infinito

PRUEBA DIRECTA (ejecutada):
   python3 << 'EOF'
   # crear sesion WHEP -> activar view request -> monitorear
   # Resultado: src=NONE en t=2 -> sesion DESAPARECE inmediatamente
   EOF
   t=1: src=1c8edf68 sessions=1  <- source cambia (orchestrador matando preview)
   t=2: src=NONE sessions=0      <- SRT cae -> MediaMTX mata sesion WHEP!
   t=3: src=NONE sessions=0
   t=4: src=c10e3e76 sessions=0  <- SRT reconecta (view mode activo)

FIX APLICADO (2026-06-06):
Archivo: /root/robiotec/dashboard/front/static/cameras_page_viewer.js

Cambio en startInferencePolling():
- ANTES: cargaba el player inmediatamente cuando online=true Y habia srcdoc
         (esto causaba el kick cycle porque durante la caida SRT hay online=false
          que luego dispara online=true -> reload con exclusive=1 -> kick)
- DESPUES: espera 3 polls CONSECUTIVOS online=true (6 segundos) antes de cargar
           el player. Esto da tiempo al modo switch (SRT reconnect ~1-2s) para
           completarse. Tambien espera 3 polls CONSECUTIVOS offline antes de
           mostrar loading screen (evita el srcdoc flip por fallo aislado).

Cambio en activateInferenceView():
- ANTES: si online=true, cargaba el player inmediatamente (antes de que el
         orchestrador haya hecho el mode switch)
- DESPUES: siempre muestra loading screen primero, deja que el polling cargue
           el player cuando el stream sea estable. Esto evita que el player
           se cree justo antes de la caida SRT.

Flujo correcto DESPUES del fix:
   t=0:  usuario activa -> loading screen -> primera poll escribe view request
   t=0-2: online=true -> consecutiveSuccess=1 (aun no llega a 3)
   t=2-3: mode switch SRT -> online=false -> consecutiveSuccess=0, consecutiveFailures=1
           (NO resetea srcdoc porque solo 1 fallo)
   t=4-8: online=true de nuevo -> consecutiveSuccess sube a 3
   t=8:   player se carga (exclusive=true) -> WHEP session creada en SRT estable
   t=13:  ICE establece (~5s) -> pce=True -> video playing!

- root user: ffmpeg re-publica las camaras de RTSP local (:8554) a SRT central
  srt://10.0.0.1:8890?streamid=publish:CAM-ID (sin /INFERENCE)

  En view mode: 1280x720@8fps (cambia ffmpeg args y reconecta SRT)
